## Problem

You are given two integers n and k.

The cost of a binary string s is defined as the sum of all indices i (0 - based) such that s[i] == '1'.

A binary string is considered valid if:

It does not contain two consecutive '1' characters.
Its cost is less than or equal to k.
Return a list of all valid binary strings of length n in any order.



Example 1:

Input: n = 3, k = 1

Output: ["000","010","100"]

Explanation:

The binary strings of length 3 without consecutive '1' characters are:

"000" : cost = 0
"100" : cost = 0
"010" : cost = 1
"001" : cost = 2
"101" : cost = 0 + 2 = 2
Among these, the strings with cost less than or equal to k = 1 are "000", "010" and "100".

Thus, the valid strings are ["000", "010", "100"].

Example 2:

Input: n = 1, k = 0

Output: ["0","1"]

Explanation:

The valid binary strings of length 1 are "0" and "1".

Thus the answer is ["0", "1"].



Constraints:

1 <= n <= 12
0 <= k <= n * (n - 1) / 2


# Approach

**Pattern used:** Backtracking / DFS with Pruning

### Core Idea

We build the binary string one position at a time.

At each index, we have two choices:

* Place `'0'`
* Place `'1'` (only if it keeps the string valid)

The backtracking explores all possible valid strings while pruning branches that would violate:

1. No consecutive `'1'` characters
2. Cost ≤ `k`

### Step-by-Step

#### State Maintained

The recursive function keeps track of:

* `index` → current position being filled
* `binary` → string built so far
* `isPreviousZero` → whether the previous character was `'0'`
* `cost` → current cost of the string

---

#### Choice 1: Place '0'

A `'0'` contributes nothing to cost and never violates the consecutive-ones rule.

So we always try:

binary + "0"

and recurse.

---

#### Choice 2: Place '1'

A `'1'` can only be placed when:

1. Previous character is not `'1'`
2. New cost remains ≤ `k`

The check:

isPreviousZero && cost + index <= k

ensures both conditions.

If valid:

* Append `'1'`
* Increase cost by `index`
* Continue recursion

---

#### Base Case

When:

index == n

a complete valid string has been constructed.

Add it to the answer list.

---

### Example

Suppose:

n = 3
k = 2

Start:

""

Index 0:

* "0"
* "1" (cost = 0)

From "0":

* "00"
* "01" (cost = 1)

From "01":

* "010"
* Cannot place another '1'

Eventually generated strings include:

000
001
010
100
101

while strings like:

011
110
111

are never explored because they violate the consecutive-ones constraint.

### Key Insights

#### 1. Cost is Built Incrementally

Instead of computing the cost at the end, we update it immediately when placing a `'1'`.

This allows early pruning.

#### 2. Consecutive Ones Are Prevented During Construction

The solution never creates an invalid prefix.

This is much better than generating all binary strings and filtering later.

#### 3. Pruning Reduces the Search Space

Whenever:

cost + index > k

the entire subtree is skipped.

No descendants can become valid because cost only increases.

### Subtle Detail

The first position can contain `'1'` because:

index = 0

and adding index `0` contributes no cost.

So:

"1..."

is always allowed provided future choices remain valid.


# Complexity

Let M be the number of valid strings generated.

**Time Complexity:** O(M × n)

* Each valid string has length `n`.
* Building and storing each output string costs O(n).
* Pruning may significantly reduce the search space.

In the worst case (large k), the number of explored states follows the Fibonacci pattern generated by the "no consecutive ones" constraint.

**Space Complexity:** O(n)

* Recursion depth is at most `n`.
* The StringBuilder stores at most `n` characters.

The returned answer list is not counted in auxiliary space.
